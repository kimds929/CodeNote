# Kilocode Mode
Kilocode(대부분 VS Code/JetBrains 계열에서 쓰는 AI 코딩 에이전트류)의 **상호작용 모드(interaction mode)** 는 “지금부터 AI가 어떤 방식으로 도와줄지”에 대한 **역할/권한/출력 스타일**을 미리 정해두는 프리셋에 가깝습니다. 같은 질문을 해도 모드에 따라:

- 무엇을 먼저 확인/질문하는지(요구사항 수집 vs 바로 작성)
- 답변 형식(설계 문서형 vs 코드 중심)
- 변경 범위(큰 리팩터링까지 제안 vs 최소 수정)
- 디버깅 접근(재현 절차/로그 요청 vs 추정 원인)
- 여러 작업을 나누고 조율하는지(오케스트레이션)

가 달라집니다.

##  시스템 프롬프트(System Prompt)의 차이
사용자가 채팅창에 질문을 입력하기 전에, Kilocode 시스템은 백그라운드에서 AI에게 매우 길고 상세한 '사전 지시사항(System Prompt)'을 먼저 전달합니다. 모드를 변경할 때마다 이 지시사항이 통째로 교체됩니다.

Architect 모드의 프롬프트 예시 (개념적):
"너는 10년 차 시니어 소프트웨어 아키텍트야. 너의 목표는 코드를 직접 짜는 것이 아니라, 사용자의 요구사항을 분석하고 최적의 시스템 구조, 데이터베이스 스키마, API 스펙을 설계하는 거야. 절대로 실제 구현 코드를 작성하지 마. 대신 마크다운(Markdown)을 사용해 다이어그램이나 구조 문서를 작성해."

Code 모드의 프롬프트 예시:
"너는 뛰어난 실무 개발자야. 아키텍트가 설계한 내용을 바탕으로 버그 없고 효율적인 코드를 작성해. 파일 생성, 수정, 터미널 명령어 실행을 적극적으로 활용해서 사용자가 지시한 기능을 실제로 작동하게 만들어."

Ask 모드의 프롬프트 예시:
"너는 친절한 코드 해설가야. 사용자의 코드를 읽고 분석해서 이해하기 쉽게 설명해 줘. 너는 시스템을 변경할 수 없어. 코드를 수정하라는 요청이 들어오면 정중히 거절하고, Code 모드로 전환하라고 안내해."

이처럼 프롬프트 자체에 역할(Role)과 금지 사항(Constraints)이 명확히 박혀 있기 때문에 AI의 말투와 접근 방식이 달라지는 것입니다.


## 도구 접근 권한(Tool Access)의 차이
| 모드 | 부여되는 도구(Tools) 목록 예시 | 결과 |
| :--- | :--- | :--- |
| **Code** | `read_file`, `write_file`, `execute_command`, `search_files` | 파일 읽고 쓰기, 터미널 실행 모두 가능 |
| **Debug** | `read_file`, `write_file`, `execute_command`, `read_logs` | 에러 수정 및 테스트 실행 가능 |
| **Architect** | `read_file`, `search_files`, `ask_user` | 파일 읽기만 가능. `write_file`이나 `execute_command` 도구 자체가 AI에게 주어지지 않음 |
| **Ask** | `read_file`, `search_files` | 읽기 전용. 시스템 변경 불가 |
| **Orchestrator** | `switch_mode`, `read_file`, `create_task` | 직접 실행 도구는 없고, 다른 모드로 전환(`switch_mode`)하는 도구를 가짐 |


## 왜 이렇게 복잡하게 나누어 놓았을까?
이러한 '프롬프트 + 도구 권한'의 분리 설계는 실무적으로 엄청난 이점을 가져옵니다.

토큰(비용) 절약 및 속도 향상: AI에게 모든 도구 사용법과 모든 역할을 한 번에 주입하면 시스템 프롬프트가 너무 길어져서 API 비용이 폭증하고 응답 속도가 느려집니다. 모드를 나누면 딱 필요한 지시사항만 전달할 수 있어 가볍고 빠릅니다.
AI의 집중력(Performance) 극대화: "설계도 하고, 코딩도 하고, 디버깅도 해"라고 지시받은 AI보다, "지금은 설계만 완벽하게 해"라고 지시받은 AI가 훨씬 더 깊이 있고 훌륭한 아키텍처를 뽑아냅니다.
안전성(Safety) 보장: Ask 모드나 Architect 모드에서는 실수로라도 운영 서버의 DB를 날리거나 중요 파일을 덮어쓰는 대참사가 발생할 수 없습니다.



아래는 질문에 나온 5개 모드를 **용도/잘 맞는 요청/대화 방식/주의점** 중심으로 정리한 설명입니다.

---

## 1) Architect 모드
### 목적
- 기능을 “코드로 바로 뛰어들기” 전에 **구조/설계/의사결정**을 정리합니다.
- 컴포넌트 분해, 데이터 흐름, API 계약, 트레이드오프 분석에 강합니다.

### 잘 맞는 상황(예시 요청)
- “결제 모듈을 어떤 구조로 나누는 게 좋을까?”
- “모놀리스를 마이크로서비스로 나누려면 경계는?”
- “이 기능을 위해 DB 스키마/테이블 설계 제안해줘”
- “캐시/큐/락 전략 비교해줘”

### 일반적인 출력 형태
- 요구사항/가정 정리 → 설계 목표(성능/확장/보안) → 아키텍처 다이어그램(텍스트) → 컴포넌트 책임 → API/이벤트 정의 → 데이터 모델 → 에러/관측성(로그/메트릭) → 단계적 구현 플랜

### 장점
- 재작업을 줄이고, 팀 커뮤니케이션(문서화)에 유리
- 복잡한 시스템에서 실수를 줄임

### 주의점/한계
- “당장 코드 한 줄”이 필요한 상황에서는 다소 느릴 수 있음
- 요구사항이 불명확하면 질문이 많아질 수 있음

---

## 2) Code 모드
### 목적
- **코드 작성/수정/리팩터링**을 빠르게 수행하는 모드입니다.
- 구현 위주의 답변(함수/클래스/파일 구조, 실제 코드 블록)이 중심입니다.

### 잘 맞는 상황(예시 요청)
- “이 스펙대로 React 컴포넌트 만들어줘”
- “이 함수 O(n)으로 바꿔줘”
- “이 API 클라이언트 코드 생성해줘”
- “테스트 코드(Jest/Pytest) 추가해줘”

### 일반적인 출력 형태
- 바로 코드 제시 + 필요한 경우 최소한의 설명
- 파일 단위 패치(어떤 파일에 무엇을 추가/변경)

### 장점
- 속도가 빠르고 생산성이 좋음
- 반복 작업(보일러플레이트/CRUD/매핑/테스트 스켈레톤)에 특히 강함

### 주의점/한계
- 큰 설계 결정을 충분히 합의하지 않은 상태에서 쓰면,
  나중에 구조가 어긋나거나 리팩터링 비용이 커질 수 있음
- “프로젝트 컨벤션/제약(레이어링, lint, 에러 처리 규칙)”을 먼저 알려주면 결과가 훨씬 안정적

---

## 3) Ask 모드
### 목적
- **질문/설명/개념 정리/Q&A** 중심입니다.
- 구현보다는 이해를 돕고, 선택지를 비교하고, 근거를 제시하는데 초점이 있습니다.

### 잘 맞는 상황(예시 요청)
- “CQRS가 뭔지, 언제 쓰는지”
- “이 에러 메시지는 어떤 의미야?”
- “Redis 캐시 전략 비교(Write-through vs Write-back)”
- “이 라이브러리 도입 장단점”

### 일반적인 출력 형태
- 개념 정의 → 핵심 포인트 → 장단점/주의점 → 예시 → 선택 가이드

### 장점
- 팀 온보딩, 기술 검토, 의사결정 문서 초안에 유용
- “왜 이 선택이 좋은지”를 정리하기 좋음

### 주의점/한계
- 바로 적용 가능한 코드 산출물이 상대적으로 적을 수 있음
- 질문이 너무 넓으면(“아키텍처 추천해줘” 정도) 범위를 좁히는 추가 질문이 필요

---

## 4) Debug 모드
### 목적
- 문제를 **재현 → 원인 후보 좁히기 → 증거 수집 → 해결/검증** 순서로 다룹니다.
- 로그, 스택트레이스, 환경정보, 재현 steps를 적극적으로 요청하는 스타일입니다.

### 잘 맞는 상황(예시 요청)
- “이 테스트만 CI에서 가끔 실패해”
- “메모리 누수 의심되는데 어디서 봐야 해?”
- “프로덕션에서 500이 뜨는데 원인을 못 찾겠어”
- “프론트에서 특정 브라우저에서만 깨져”

### 일반적인 진행 방식
1) 증상 정리(언제/어디서/빈도/최근 변경)
2) 재현 방법/최소 재현 코드 요청
3) 관측(로그 레벨, metric, profiler, trace) 제안
4) 원인 가설 나열 → 각 가설을 검증하는 체크리스트
5) 수정안 제시 + 회귀 테스트/재발 방지책

### 장점
- “추측으로 코드 바꾸기”를 줄이고, 체계적으로 해결 가능
- 재발 방지(테스트/모니터링/가드레일)까지 이어지기 쉬움

### 주의점/한계
- 제공 정보(로그/환경)가 부족하면 정확도가 떨어짐
- 보안/개인정보가 포함된 로그는 마스킹해서 공유해야 함

---

## 5) Orchestrator 모드 (질문에 적힌 Orchastrator는 보통 Orchestrator의 오타로 보입니다)
### 목적
- 여러 작업을 **쪼개서(task decomposition) 순서를 정하고**, 필요한 경우 각 작업에 적절한 모드/단계를 배정해 “프로젝트처럼” 진행합니다.
- 큰 작업(기능 단위 개발, 마이그레이션, 릴리즈 준비)에 적합합니다.

### 잘 맞는 상황(예시 요청)
- “회원가입~이메일 인증~프로필 설정까지 한 번에 구현 계획 잡아줘”
- “레거시 API를 v2로 마이그레이션하는 단계별 플랜”
- “관측성(로그/메트릭/트레이싱) 도입을 로드맵으로”
- “프론트+백엔드+DB 변경이 함께 필요한 기능”

### 일반적인 출력 형태
- 목표/범위 확정 → 작업 목록(WBS) → 우선순위/의존성 → 단계별 산출물 → 위험요소/롤백 전략 → 체크리스트(테스트, 배포)

### 장점
- 한 번에 많은 걸 해야 할 때 길을 잃지 않게 해줌
- 팀 작업 분담/이슈 쪼개기에 바로 사용 가능

### 주의점/한계
- “지금 당장 코드 10줄” 같은 단발성에는 오버킬일 수 있음
- 프로젝트 상황(인력/기간/기술 스택/배포 방식)을 알려줘야 계획 품질이 좋아짐

---

# 모드 선택 가이드(빠르게)
- **설계/구조/경계/의사결정**: Architect  
- **바로 구현/리팩터링/테스트 작성**: Code  
- **개념 질문/비교/학습**: Ask  
- **오류 원인 추적/성능 이슈/재현 기반 해결**: Debug  
- **큰 작업을 쪼개고 단계·역할·순서를 정리**: Orchestrator  

---

원하면, 당신이 현재 하려는 작업(기능/스택/리포 구조/제약)을 4~5줄로 알려주면 “어떤 모드로 시작하고, 중간에 어떤 모드로 전환하면 효율적인지”까지 작업 흐름으로 추천해줄게요.
